## Acceptance Criteria

1. **Availability Slots for All Mentors:**

- Each mentor has 3 availability blocks across upcoming 3 weeks
- Week 1: Random date NOW() + random {1-7} days, random time 8am-8pm, random duration (15min, 30min, 60min)
- Week 2: Random date NOW() + random {8-14} days, random time 8am-8pm, random duration (15min, 30min, 60min)
- Week 3: Random date NOW() + random {15-21} days, random time 8am-8pm, random duration (15min, 30min, 60min)
- SQL seed file uses dynamic date calculation (e.g., `NOW() + INTERVAL '5 days' + INTERVAL '9 hours'`)
- Each availability block generates time slots (per existing availability schema)

1. **Booking Slots for Mentees (demand-driven):**

- Each mentee books a small, realistic number of meetings (random 3-8 across the 3-week window), rather than filling a
  fixed share of mentor slots. Most mentor slots stay open.
- Status mix: ~20% confirmed, ~80% pending (most requests await mentor action)
- Confirmed requests carry a `confirmed_at` randomly between the request's `created_at` and now; pending requests have
  none
- Each request's `created_at` is back-dated within its 7-day liveness window (a pending request auto-rejects after 7
  days), not stamped at a single seed instant
- Random meeting_goal text (from predefined templates: "Discuss fundraising strategy", "Get feedback on product
  roadmap", etc.)

1. **Test Coverage:**

- Test: Verify each mentor has 3 availability blocks and their generated time slots
- Test: Verify bookings are demand-driven (3-8 per mentee), not a fixed share of slots
- Test: Verify booking status distribution (~20% confirmed, ~80% pending)
- Test: Verify confirmed bookings have a confirmed_at and pending bookings do not

1. **Production Seed Files Updated:**

- All changes applied to `supabase/seeds/production/*.sql` to match local seed behavior
- Production seeds use same dynamic date logic for realistic test data in production environment

## Seed shape notes

- **User pool:** mentors represent accelerator alumni (a large pool); mentees are the current and recent cohorts (a
  smaller pool). The realistic ratio carries more mentees than this seed currently holds — roughly twice the present
  count is the target — and growing the mentee pool is follow-up data work.
- **Booking rate is demand-driven, not supply-driven.** Bookings scale with mentee demand (a few per mentee), not as a
  share of mentor supply. The original "~50% of all slots" rule over-subscribes the small mentee pool (hundreds of
  alumni mentors × three blocks each, divided across a few dozen mentees, yields ~100+ bookings per mentee).
- **Baked into the seed, refreshed on a schedule.** `scripts/convert_backup_to_d1.py` appends a self-correcting footer
  to the generated seed: it thins bookings to the demand-driven per-mentee shape, sets the status mix, anchors every
  timestamp to load time, and back-dates each request's `created_at` / `confirmed_at`. A freshly loaded database is
  therefore correct with no manual step. The date portion (`scripts/bump-seed-dates.sql`) also runs weekly via
  `.github/workflows/bump-seed-dates.yml` to keep an already-loaded demo current.

## Dev Notes

### Story Context

This story enhances Story 0.2 (SKEL-DB-002: Mock Data Seeding Script) by adding realistic availability and booking data
for local development. The original seed script created users, taxonomy, and portfolio companies, but lacked
availability slots and bookings needed for testing booking flows.

**Why This Matters:**

- Developers need realistic test data to work on booking features locally
- Without availability slots, mentees can't test browsing/booking flows
- Without existing bookings, dashboard and calendar features show empty states
- Dynamic date generation ensures test data is always "upcoming" (not expired)
- Separate availability and booking seed files allow independent re-seeding

**Story Boundaries:**

- ✅ Create seed data for availability blocks and bookings
- ✅ Update production seed files to match (same dynamic date logic)
- ✅ Fix user seed files for local development (ETL disabled for users)
- ✅ Separate availability and booking generation for clarity
- ❌ NO changes to database schema (already defined in migrations)
- ❌ NO changes to API or frontend (this is pure data seeding)
- ❌ NO production email updates (special emails not provided)

### Seed Data Shape

Each mentor has three availability blocks (1-3 hour windows, subdivided into 15/30/60-minute slots) across the upcoming
three weeks. Bookings follow the demand-driven shape applied by the post-seed correction scripts (a few per mentee, ~20%
confirmed / ~80% pending), with every date anchored to the current date. Exact totals vary per seed run, so verify the
shape with the queries in this story rather than fixed counts.
